# TASK-0019 — MutationEngine Hardening + Core Integration Fix

## Goal

Make MutationEngine outcomes **machine-verifiable** and eliminate **bypass paths** and **truthiness bugs** in Core. Align MutationFlow with existing governance patterns: TrustDecision + ReviewQueue + ApprovalStore + explicit exceptions.

## Required fixes

### 1) Replace string-return outcomes with explicit exceptions + structured result

**Problem:** `MutationEngine.apply()` and related methods return strings that callers can't reliably interpret.

**Requirement:**

* Introduce explicit exception types (in `mutation.py`), similar to `TrustDeniedError`:
  * `MutationDeniedError` (e.g., protected path without override)
  * `MutationReviewRequiredError` (when TrustDecision is "review" for governance mutation override)
  * `MutationApplyError` (unexpected failure applying patch)
* On success, return a small structured object (dataclass ok) like:
  * `MutationResult(ok: bool, changed_files: list[str], backup_paths: list[str], summary: str)`
* Do **not** return plain strings for success/denial/failure.

**Acceptance check:** Callers can distinguish success/deny/review/failure without parsing text.

### 2) Remove or quarantine `review_with_gpt()` (bypass)

**Problem:** `review_with_gpt()` performs network calls and bypasses TrustMatrix/ReviewQueue.

**Requirement (choose A, do not do both):**

* **A (preferred):** Remove `review_with_gpt()` usage from the mutation path entirely. Keep the function only if it is unused and clearly marked "deprecated/disabled" with a comment.
* **B:** If it must remain callable, it must:
  * route through existing network gateway (WebReader) or a new AI gateway (NOT allowed in this task)
  * and require TrustMatrix gating + review queue

Given scope restrictions, **do A**.

**Acceptance check:** There is no reachable code path in MutationEngine that performs direct network calls.

### 3) Fix Core truthiness bug and normalize action usage

**Problem in `core.py`:**

* `validate_trust_for_action()` returns `TrustDecision`, but Core treats it like a boolean.
* Core uses `"mutation"` string instead of normalized constants.

**Requirement:**

* Replace any Core logic like:
  * `if not self.trust.validate_trust_for_action(...):`
    with decision-aware logic:
  * `decision = self.trust.validate_trust_for_action(...); if decision.decision == "deny": ...; if "review": ...; if "allow": ...`
* For mutation-related gating, use **normalized constants** from `trust.py`:
  * `GOVERNANCE_MUTATION` when overriding protected paths
* If Core triggers mutation proposals that are not governance mutations, define the correct action constant or document as legacy and add guardrail (do not invent new ad-hoc strings).

**Acceptance check:** No Core path uses TrustDecision as bool. No mutation gating uses raw `"mutation"` string.

### 4) Align governance override flow with ReviewQueue + ApprovalStore

**Requirement:**

* If mutation touches protected paths and `allow_governance_mutation=False`:
  * raise `MutationDeniedError`
* If `allow_governance_mutation=True`:
  * Call `validate_trust_for_action(..., GOVERNANCE_MUTATION, context={touched_paths,...})`
  * If decision == `"deny"`: raise `MutationDeniedError`
  * If decision == `"review"`:
    * enqueue a review request (same pattern as gateways)
    * raise `MutationReviewRequiredError(request_id=...)`
  * If decision == `"allow"`:
    * proceed
* Support replay approval (same pattern as gateways):
  * if `request_id` is provided and approved with matching context, bypass review/deny as appropriate.
  * Context must include touched_paths list (sorted) for hashing consistency.

**Acceptance check:** governance mutation override uses the same review/approve/replay workflow as gateways.

### 5) Add behavioral smoke tests: `tests/test_mutation_smoke.py`

Must include deterministic tests:

A) **Protected path denied**
* Attempt mutation touching `CONTROL.md` without override
* Expect `MutationDeniedError`

B) **Protected path review**
* Force TrustMatrix to return `review` for `GOVERNANCE_MUTATION`
* Expect `MutationReviewRequiredError` and review_queue entry

C) **Approval replay works**
* Approve request_id with matching context
* Re-run apply with request_id
* Expect success (MutationResult.ok True) without re-enqueueing

D) **Core uses decision semantics**
* Unit test a Core mutation entrypoint (or minimal method) to confirm it branches on `decision.decision` not truthiness

If Core has no testable mutation entrypoint, add a minimal non-invasive method only if necessary (report if you must).
