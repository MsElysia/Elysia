# TASK-0044 — Artifact Side-Effect Policy + Tests (REPORTS/ History / ApprovalStore / Queue)

## Goal

Define and enforce a clear policy for which **artifacts** may be written under `REPORTS/` (and related files like approval store / review queue) during:

* deny
* review
* allow / success

Then add tests to lock the policy so “no side-effects” is unambiguous.

This is not feature work; it’s governance + determinism.

## Policy

### A) Always allowed artifacts (any run, including deny/review)

These are allowed to be written because they’re observability, not execution:

* `REPORTS/run_once_last.json`
* `REPORTS/run_once_history/*` (if history retention is enabled)
* `REPORTS/acceptance_last.json` and `acceptance_last.log` **ONLY** when acceptance actually runs

### B) Forbidden artifacts on deny/review (must NOT be written)

During deny or review outcomes (no “allow” execution), the system must not create/modify:

* Any mutation backup directories/files (e.g., `guardian_backups/*`)
* Any changed target files
* Any `REPORTS/acceptance_last.*` or acceptance history files
* Any `REPORTS/subprocess_background.jsonl` entries
* `REPORTS/approval_store.json` (must not change unless an explicit approve/deny action is taken via UI/endpoint)

### C) Allowed artifact on review outcome

On “review required” outcomes, it is allowed to:

* Append to `REPORTS/review_queue.jsonl` (enqueue request)

But it must not:

* Write approval store
* Write backups
* Start subprocess/network

### D) Allowed artifacts on allow/success

On allow/success, permitted:

* Backups created by MutationEngine
* Target files changed by mutation
* Subprocess audit line if background run actually launched
* Acceptance artifacts/history if acceptance executed
