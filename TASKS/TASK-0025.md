# TASK-0025 — Unify Acceptance Execution via SubprocessRunner (Single Subprocess Surface)

## Goal

Eliminate direct `subprocess.run` usage from Core by routing acceptance execution through `SubprocessRunner`. This enforces the "all external power goes through gateways" doctrine and prevents drift.

## Scope (allowed files)

* `project_guardian/core.py`
* `project_guardian/subprocess_runner.py` (only if a missing hook prevents safe acceptance execution)
* `project_guardian/trust.py` (constants/imports only; no policy changes unless required for correctness)
* `tests/test_task_execution.py` (update)
* `tests/test_gateways_smoke.py` (update only if needed)
* `SPEC_MODULES/core.md` (update)
* `SPEC_MODULES/gateways.md` (update)
* `REPORTS/module_audit_core.md` (update)
* `REPORTS/module_audit_gateways.md` (update)
* `TASKS/TASK-0025.md` (new)
* `CHANGELOG.md`
* `REPORTS/AGENT_REPORT.md`
* `CONTROL.md`

## Non-goals

* No UI changes
* No new dependencies
* No change to whitelisted task types
* No arbitrary command support

## Requirements

### 1) Core must not call subprocess directly

Search in `project_guardian/core.py` for any direct subprocess usage:
* `subprocess.run`
* `subprocess.Popen`
* `os.system`
* etc.

Replace acceptance execution with `SubprocessRunner.run_command(...)`.

### 2) Initialize SubprocessRunner in Core

Ensure Core constructs a single SubprocessRunner instance with:
* shared TrustMatrix
* ReviewQueue
* ApprovalStore
* memory (if available)

Use the same injection pattern used for WebReader.

If SubprocessRunner is created on demand, standardize it so it is not duplicated and always uses the same Trust and stores.

### 3) Acceptance execution policy (strict)

Acceptance execution must remain limited to the known safe command only:
* Powershell: `.\scripts\acceptance.ps1`

Implementation guidance:
* In Core's `_execute_run_acceptance()` build a fixed command list (no user-controlled input), e.g.:
  * `["powershell", "-NoProfile", "-ExecutionPolicy", "Bypass", "-File", "<ABS_PATH_TO_ACCEPTANCE_PS1>"]`
* Call `subprocess_runner.run_command(...)` with:
  * `timeout=300` seconds (5 minutes)
  * `request_id` optional (only if review workflow applies)

### 4) Trust action semantics

SubprocessRunner uses the action constant `SUBPROCESS_EXECUTION`.
Core must not reintroduce ad-hoc strings.
If SubprocessRunner currently enforces review/deny, preserve that behavior.

**Important:** Running acceptance should not require manual approvals in normal dev flow unless your TrustMatrix policy dictates it. If it currently reviews/denies by default, document the behavior and ensure tests mock allow.

### 5) Structured result preservation

Core's RUN_ACCEPTANCE task result must stay:
```json
{ "status":"ok", "task_id":"...", "outcome":"acceptance_ran", "exit_code": <int> }
```

If SubprocessRunner returns stdout/stderr, you may ignore or attach under optional keys. Do not change required keys.

## Tests

### Update `tests/test_task_execution.py`

* Ensure RUN_ACCEPTANCE no longer patches `subprocess.run` in Core.
* Patch/mock `SubprocessRunner.run_command` instead.
* Assert it is invoked with the expected fixed command and timeout.
* Ensure structured result still matches.

### Optional: Update gateway tests only if necessary

If `tests/test_gateways_smoke.py` assumes SubprocessRunner is unused by Core, adjust accordingly.
