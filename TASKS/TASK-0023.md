# TASK-0023 — APPLY_MUTATION Task Type (Payload File in MUTATIONS/)

## Goal

Add a new whitelisted task type `APPLY_MUTATION` that executes MutationEngine using a **separate payload file** under a controlled directory (`MUTATIONS/`). Support deny/review/approve/replay flows with structured results. No freeform markdown interpretation.

## Task contract format (strict)

A task file `TASKS/TASK-XXXX.md` with exactly these directives (order doesn't matter, but all required unless stated):

* `TASK_TYPE: APPLY_MUTATION`  (required)
* `MUTATION_FILE: MUTATIONS/<name>.json` (required; must be under MUTATIONS/)
* `ALLOW_GOVERNANCE_MUTATION: true|false` (required)
* `REQUEST_ID: <id>` (optional; used for approval replay)

Validation rules:
* Exactly one `TASK_TYPE` directive.
* `MUTATION_FILE` must:
  * start with `MUTATIONS/`
  * end with `.json`
  * contain no `..` segments
* If validation fails: structured error:
```json
{ "status":"error", "code":"TASK_CONTRACT_INVALID", "task_id":"TASK-XXXX", "detail":"..." }
```

## Mutation payload format (`MUTATIONS/*.json`) (strict)

JSON with:
```json
{
  "touched_paths": ["project_guardian/some_file.py", "..."],
  "changes": [
    { "path": "project_guardian/some_file.py", "content": "FULL_FILE_CONTENTS_STRING" }
  ],
  "summary": "short human summary"
}
```

Rules:
* `touched_paths` must match the set of `changes[].path` (same elements, order ignored).
* All `path` values must be relative, no `..`, no absolute paths.
* Hard block attempts to touch anything outside repo root.
* If invalid: structured error `MUTATION_PAYLOAD_INVALID`.

Note: This uses "full file replacement" semantics (as documented gap). Do not implement patch/diff in this task.

## Core execution behavior

### 1) Extend whitelist

Add `APPLY_MUTATION` to Core's whitelisted task types.

### 2) Implement `_execute_apply_mutation(task_id, contract_directives)`

Steps:
1. Load and validate `MUTATION_FILE`
2. Parse JSON payload and validate schema + path safety
3. Build mutation context for governance decision/replay:
   * `context = { "touched_paths": sorted(touched_paths), "override_flag": allow_governance_mutation, "caller":"GuardianCore", "task_id": task_id }`
4. Call MutationEngine apply with:
   * `allow_governance_mutation` from directive
   * pass `request_id` if provided (for replay)
   * pass caller identity/task_id if supported; otherwise include in context where MutationEngine already uses it
5. Handle outcomes:
   * Success → return structured result with changed_files, backup_paths, summary
   * `MutationReviewRequiredError` → return `{status:"needs_review", request_id, ...}`
   * `MutationDeniedError` → return `{status:"denied", code:"MUTATION_DENIED", ...}`
   * `MutationApplyError` → return `{status:"error", code:"MUTATION_APPLY_FAILED", ...}`

### 3) Ensure `run_once()` executes APPLY_MUTATION when ready

No change to routing logic beyond calling executor when "ready".

## Tests — `tests/test_apply_mutation_task.py` (deterministic)

Use `tmp_path` and monkeypatch Core to point to temp CONTROL/TASKS/MUTATIONS dirs.

Required tests:
1. Invalid contract (bad MUTATION_FILE path) → TASK_CONTRACT_INVALID
2. Invalid payload (touched_paths mismatch) → MUTATION_PAYLOAD_INVALID
3. Protected path without override → expect status=="denied"
4. Review flow → expect status=="needs_review" and request_id present
5. Approval replay → expect status=="ok" and mutation applied

Do not perform real repo mutations in tests; use temp dirs.
