# TASK-0039 — Background Subprocess Audit Trail (Append-Only JSONL + Minimal Tests)

## Goal

Add an **append-only audit log** for background subprocess launches so they are observable, durable, and attributable (task_id / request_id / caller). This reduces the "fire-and-forget" risk introduced by background mode.

## Scope (allowed files)

* `project_guardian/subprocess_runner.py`
* `project_guardian/ui/app.py` (read-only; no UI changes in this task unless absolutely necessary)
* `tests/test_subprocess_background_audit.py` (new)
* `SPEC_MODULES/gateways.md` (update)
* `REPORTS/module_audit_gateways.md` (update)
* `TASKS/TASK-0039.md` (new)
* `CHANGELOG.md`
* `REPORTS/AGENT_REPORT.md`
* `CONTROL.md`

### Non-goals

* No process supervision
* No killing/stopping processes
* No stdout/stderr streaming
* No scheduler/daemon
* No new dependencies

## Requirements

### 1) Append-only audit log file

Create an audit log file at:

* `REPORTS/subprocess_background.jsonl`

Each background launch appends exactly one JSON line with fields:

Required:
* `ts`: ISO8601 timestamp (UTC preferred, but consistent)
* `pid`: int
* `command`: list[str] (redacted as needed; see below)
* `cwd`: str | null
* `timeout_s`: int | null (likely null for background)
* `caller_identity`: str | "unknown"
* `task_id`: str | "unknown"
* `request_id`: str | null
* `action`: should be the constant for subprocess execution (e.g., `SUBPROCESS_EXECUTION`)
* `decision`: `"allow"` (background runs only occur when allowed or replay-approved)
* `context_hash`: optional (if you already compute one; otherwise omit)
* `notes`: optional string (e.g., "background")

Redaction rules:
* If any command arg contains obvious secrets (token/key/secret/password) redact that arg to `"***REDACTED***"`.
* If you cannot robustly detect, implement minimal keyword-based redaction (deterministic).

Durability:
* Ensure `REPORTS/` exists (best effort).
* Append should be atomic in practice (single-writer). Use `open(..., "a", encoding="utf-8")` and write one line + `\n`, flush.

Failure behavior:
* If audit write fails, do **not** crash the subprocess call. Log best-effort (if there is a memory logger) or silently continue. This is an audit enhancement, not a runtime blocker.

### 2) Integrate with background execution

In `SubprocessRunner.run_command_background(...)` (or whatever the method is named):

* After the process is successfully started (PID known), append the audit record.
* Include the exact `caller_identity`, `task_id`, `request_id` that were passed.
* Ensure the record is written only when the action actually launched.

### 3) Ensure review/deny paths do not write audit lines

* Deny: no audit line
* Review: no audit line
* Replay-approved allow: audit line should still be written (decision "allow", include request_id)

### 4) Spec + audit docs

Update `SPEC_MODULES/gateways.md`:
* Describe background audit log format and location.

Update `REPORTS/module_audit_gateways.md`:
* Add note that background runs are now auditable via JSONL.

## Tests — `tests/test_subprocess_background_audit.py`

Use `tmp_path` and monkeypatch the REPORTS path (or module constant) so tests don't touch real filesystem.

Must test:

1. `test_background_launch_writes_audit_line()`
   * Mock `subprocess.Popen` to return object with `pid=1234`
   * Run background command with allow decision
   * Assert file exists
   * Assert last line parses as JSON
   * Assert contains pid, command list, caller_identity/task_id fields

2. `test_review_or_deny_does_not_write_audit()`
   * Force TrustMatrix to return deny or review
   * Ensure file not created or remains empty

3. `test_redaction_applies_to_command_args()`
   * Provide command args containing "token=abc"
   * Assert written JSON has redacted arg(s)

If path plumbing is hard, implement a helper in SubprocessRunner to compute audit path that can be monkeypatched.
