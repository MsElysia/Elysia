# TASK-0026 — Control Panel MVP Expansion (Tasks + Run Once + Mutation Builder)

## Goal

Expand the existing FastAPI + Jinja control panel so it can:
1. Display Core status (including last `run_once()` result)
2. Trigger `run_once()` safely
3. Create whitelisted task files from templates (RUN_ACCEPTANCE, CLEAR_CURRENT_TASK, APPLY_MUTATION)
4. Create and save mutation payload JSON files under `MUTATIONS/` with strict validation

All operations must remain **local-only** and must not introduce arbitrary execution or new external action surfaces.

* No network calls other than serving localhost UI
* No new dependencies unless absolutely required (prefer none)

## Required UI Features

### 1) Dashboard additions

On `/` dashboard:
* Show:
  * `CURRENT_TASK` from CONTROL.md
  * pending review count
  * last acceptance status from `REPORTS/acceptance_last.json`
  * **last run_once result** (new artifact; see below)
* Add buttons:
  * "Run Once"
  * "Run Acceptance" (creates RUN_ACCEPTANCE task and sets CURRENT_TASK to it, OR calls existing endpoint if you already have one)
  * "Clear Current Task"
  * "Create Task" (go to Task Builder)
  * "Create Mutation Payload" (go to Mutation Builder)

### 2) Run Once endpoint (safe)

Add:
* `POST /control/run-once`
  Behavior:
* Instantiate GuardianCore in-process (same as you do elsewhere)
* Call `core.run_once()`
* Write artifact:
  * `REPORTS/run_once_last.json` (timestamp + full structured result)
* Redirect back to dashboard.

Safety:
* Must not allow arbitrary args.
* Must not execute tasks other than what CONTROL.md points to (run_once already governs this).

### 3) Task Builder UI (new)

Add page:
* `GET /tasks/new` → `task_builder.html`
* Form options:
  * Task ID (default `TASK-UI-<timestamp>` or similar safe scheme)
  * Task type dropdown: `RUN_ACCEPTANCE`, `CLEAR_CURRENT_TASK`, `APPLY_MUTATION`
  * For APPLY_MUTATION:
    * Mutation file path selector (must be `MUTATIONS/*.json`)
    * allow_governance_mutation checkbox
    * optional request_id field
* On submit:
  * `POST /tasks/create`
  * Validate strictly:
    * task_id matches `^TASK-[A-Za-z0-9_-]{1,32}$` (or your existing style)
    * write file `TASKS/<task_id>.md` with strict directives only (one TASK_TYPE line + required lines)
  * Option to also set CONTROL.md CURRENT_TASK to this new task (checkbox "Activate now").

### 4) Mutation Payload Builder UI (new)

Add page:
* `GET /mutations/new` → `mutation_builder.html`
* Form fields:
  * payload name (creates `MUTATIONS/<name>.json`)
  * summary
  * one or more file changes:
    * path (relative)
    * content (textarea full file contents)
* On submit:
  * `POST /mutations/create`
    Validation (must match TASK-0023/0024 rules):
* output JSON schema:
```json
{
  "touched_paths": [...],
  "changes": [{"path":"...", "content":"..."}],
  "summary":"..."
}
```
* `touched_paths` must match `changes[].path` set
* path safety:
  * no `..`
  * no absolute paths
  * must be within repo root
* Write under `MUTATIONS/` only.

### 5) Reviews pages remain intact

No regressions for:
* list reviews
* view detail
* approve/deny
  Make sure approve/deny still updates ApprovalStore and ReviewQueue correctly.

## Data/artifacts

* Read:
  * `REPORTS/acceptance_last.json`
  * `REPORTS/acceptance_last.log`
* Write:
  * `REPORTS/run_once_last.json`
  * new task files in `TASKS/`
  * mutation payloads in `MUTATIONS/`

All writes must be atomic (tmp + replace) consistent with prior hardening.

## Tests (preferred minimal)

Add `tests/test_ui_smoke.py` that uses FastAPI TestClient:
* GET `/` returns 200
* POST `/control/run-once` returns redirect (302) and writes run_once artifact (use temp dirs / monkeypatch paths)
* POST `/tasks/create` with invalid data returns 400
* POST `/mutations/create` with invalid paths returns 400

If TestClient introduces complexity due to file paths, document it and skip tests, but prefer to include at least one.
