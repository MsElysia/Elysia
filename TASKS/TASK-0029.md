# TASK-0029 — Mutation Payload Base Hashing + Diff Mismatch Warnings

## Goal

Make mutation diffs trustworthy by capturing a **base hash per file** at payload creation time and showing **mismatch warnings** in the diff viewer when the current file differs from the recorded base.

## Scope (allowed files)

* `project_guardian/ui/app.py`
* Templates:
  * `project_guardian/ui/templates/mutation_builder.html` (update)
  * `project_guardian/ui/templates/mutation_detail.html` (update)
  * `project_guardian/ui/templates/diff_viewer.html` (update)
  * `project_guardian/ui/templates/mutations.html` (optional)
* `tests/test_ui_diff_basehash.py` (new)

## Required changes

### 1) Extend mutation payload schema (backward compatible)

When creating a payload in `POST /mutations/create`, add:

```json
{
  "touched_paths": [...],
  "changes": [{"path":"...", "content":"..."}],
  "summary": "...",
  "base": {
    "project_guardian/foo.py": {
      "sha256": "<hex>|MISSING",
      "bytes": <int>|0,
      "captured_at": "ISO8601"
    }
  }
}
```

Rules:
* `base` is optional for older payloads; UI must handle missing `base`.
* For each change path:
  * If current file exists: compute sha256 of bytes and store bytes length.
  * If missing: store `"sha256": "MISSING", "bytes": 0`.
* `captured_at` once per file OR global timestamp (either acceptable, but be consistent).

### 2) Diff viewer mismatch warnings

Update `GET /diff?mutation=<file>&path=<path>`:

Current behavior compares:
* proposed (payload content)
* current file content

Enhancements:
* Load base info from payload (if present)
* Compute current sha256 (or MISSING)
* If base exists and base.sha256 != current.sha256:
  * Display a warning banner:
    * "Base mismatch: file changed since payload creation"
    * show base sha256 and current sha256
    * show captured_at
* If base missing: show "No base recorded (legacy payload)."

### 3) Optional: "Diff vs base" mode

Add query param:
* `/diff?...&mode=current|base` (default current)
  If `mode=base` and base exists and base.sha256 != "MISSING":
* Compare proposed vs base content (but you don't have base content).
  Since base content wasn't stored, DO NOT implement this unless you also store base content.
  So default plan:
* **Do NOT add diff-vs-base** unless you decide to store base content too (not required).

### 4) Update mutation detail page

In `GET /mutations/{filename}` detail view:
* Show for each touched path:
  * base sha (or "legacy/no base")
  * current sha
  * indicator: MATCH / MISMATCH / MISSING
* Provide a "View Diff" link (existing)

### 5) Safety constraints

* Continue strict filename/path validation
* Do not allow arbitrary file reads:
  * path must be in touched_paths
* Limit any displayed hashes/strings to safe escaped output

## Tests — `tests/test_ui_diff_basehash.py`

Use TestClient + tmp_path with monkeypatch of repo root / MUTATIONS dir.

Required tests:

1. Payload creation includes base hashes
   * Create a file in temp repo root
   * Create payload via POST
   * Assert payload JSON contains base[path].sha256 matching computed hash

2. Diff viewer shows mismatch warning
   * Create payload when file content = A
   * Modify file to content = B
   * Request diff page
   * Assert response contains "Base mismatch" (or exact banner text you implement)

3. Legacy payload handling
   * Create a payload JSON without base
   * Diff endpoint should return 200 and show "No base recorded" (or similar)
